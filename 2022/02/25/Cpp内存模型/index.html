C++：内存模型

一. 内存模型

程序在运行时，OS为该进程分配一块内存，具有以下结构：

注：底部为高地址

  • 文本/代码段：保存执行指令，通常可共享，只会有一个副本。
  • 数据段
    • 初始化数据段：存储未初始化的全局和静态变量，此段可读可写
    • 未初始化数据：常被称作 BSS segment，由内核在程序执行前初始化为0，其位于初始化数据段的后方。
  • 栈：临时变量以及虚指针，在栈中分配变量，地址会依次递减，向低地址拓展
    • 栈帧：函数调用所操作的部分，每个函数调用都有独立的栈帧结构
  • 堆：动态内存分配

全局和静态变量的初始化都发生在main()调用之前，因此可以借此在之前实现一些东西。下面这个在GCC环境下也可以

Stack

栈顶地址总是小于栈的基地址。

  • windows API中_stdcall，由被调函数调整堆栈。
  • ANSI C中_cdecl，则是调用者调整堆栈。
  1. 先将func的三个参数按照从右往左的顺序入栈，ESP–
  2. 然后压入返回地址（RET）。ESP–
  3. 将当前帧指针(EBP)入栈
  4. 调整EBP和ESP，为func函数创建栈帧结构
  5. func调用返回之前，恢复堆栈（ESP=ESP+3*4）

__fastcall：快速调用，一部分参数直接放在寄存器（ECX&EDX），剩下的参数从右向左传

Windows下，由编译器决定堆栈的大小，默认都为1MB

Heap

下面是和堆相关的API

进程初始化时，系统会自动为进程创建默认堆，所占1M。堆对象由系统进行管理。

hHeap就指向HEAP首部的地址，进程的用户区存在PEB（进程环境块），这个结构存放这有关进程的信息，其首地址偏移ox18是进程默认堆地址，0x90则存放指向进程所有堆的地址列表的指针。同一时刻只能由一个线程访问堆数据，多线程访问同一个堆的话，只能排队等候。

二. 类的内存分布

  • Stack: area,perimeter,_vfptr
  • TextSeg:Shape(),~Shape(),GetArea(),GetPerimeter(),static Shape::GetPI
  • DataSeg：static Shape::PI

总结：

  • 函数无论静态与否，都放在文本段
  • 普通变量放在栈，静态变量放在数据段
  • 虚指针在栈中

最核心的便是{ % label 虚表指针 + 数据成员 + 内存对齐 pink %}

说一下内存对齐：

CPU访存拥有一个细粒度，如果一个数据在一个细粒度的两侧，那么CPU额外需要访问两次，并对得到的数据进行删减拼接，就会产生性能的浪费。

内存对齐有以下几个规则：

  • class/struct的对齐值由其最大的成员决定。可以通过alignof关键字查看
  • 可以给编译器指定一个最大的对齐值，g++下：#pragma pack(n)
  • 实际对齐由上面两个中最小的决定
  • 空类/结构体大小为1：编译器添加的char，使其对象不同实体具有独特的地址

成员数据

单继承

在单继承下，子类只是将一个父类的副本放在自己内存的首部（低地址->高地址）。char将其填充为4byte

多继承

和单继承情况类似，使用非虚多继承时，将两个父类副本按照继承的顺序放置在内存首部。

菱形继承

编译器为每个直接基类都添加了一个vbptr（虚基类表），直接基类依旧按照固有顺序，虚基类则沉底。以DerivedA的虚表为例，此时一共有两项：

  1. DerivedA对象指针和其vbptr之间的偏移

总结以下：

  • 首先排列非虚继承的基类实例
  • 有虚基类时，为每一个基类添加一个vbptr
  • 排列派生类数据成员
  • 在最后放置基类成员

成员函数

不占用类实例的空间，编译器为每一个成员函数和类做对应工作。每个成员函数编译器都会为其分配一个对应的类类型对象this已经一个别名。同样，this指针也是通过偏移访问成员变量，在没有虚基类的情况下，如果编译器把this指针缓存到了寄存器中，访问成员变量的过程将与访问局部变量的开销相似。

成员函数也可以被覆盖，根据是否添加关键字virtual来区分编译时覆盖和运行时覆盖。

  • 编译期覆盖：调用根据左边的变量类型决定
  • 运行时：只能有指针调用，根据实际类型来决定。

为了实现这种机制，引入了隐藏的vfptr成员变量。一个vfptr被加入到类中（如果类中没有的话），该vfptr指向类的虚函数表（vftable）。类中每个虚函数在该类的虚函数表中都占据一项。每项保存一个对于该类适用的虚函数的地址。因此，调用虚函数的过程如下：取得实例的vfptr；通过vfptr得到虚函数表的一项；通过虚函数表该项的函数地址间接调用虚函数。也就是说，在普通函数调用的参数传递、调用、返回指令开销外，虚函数调用还需要额外的开销。在一种实现中，一个体系内的类只会拥有一个vfptr，其从基类继承。当派生类添加新的虚函数时，只在原有的表中增添新项。

虚函数在构造函数调用时绑定，构造函数将vptr指向对应的vtab，所以不要再构造函数中调用虚函数，会出现问题。

结果是输出两次“I really am Erdos”。

  • 第一次是创建Erdos对象时调用构造，设置虚表，调用函数，没有问题。
  • 第二次出现问题：创建Fermat对象时，要先调用Erdos的构造函数，又因为虚表在构造函数调用时绑定，所以在Fermat中写的虚函数并不会被调用。

这两行语句的调用结果一致，最终都是调用DerivedC::AVF,但是实际过程不同。

  • ((DerivedA*)pc)->AVF()->(DerivedA)pc->DerivedA::vfptr[0]((DerivedC*)(DerivedA*)pc,也就是访问DerivedA的虚函数表，取出第一项，然后将pc转化成DerivedC*再作为this指针传进去，由于A为C的最先继承类，二者地址是没有偏移的。
  • 但是DerivedB作为第二个基类（非最先继承类），其地址存在偏移，为了正确转换，其虚表存在着一个调整块，&thunk:this-=16，DerivedB向前调整16byte则为DerivedC,之后再进行调用。在微软VC++实现中，对于有虚函数的多重继承，只有当派生类虚函数覆盖了多个基类的虚函数时，才使用调整块。

接下来考虑另一个函数DerivedC::BVF()，其同样是非最先继承类，一般的实现是同样使用调整块。MSVC中，则将其编译为一个接收一个指向DerivedC中嵌套的DerivedB实例。

总结就是，再MSVC中，对于有虚函数的多继承进行以下的处理：

  • 派生类覆盖了多个基类的同名虚函数，创建调整块
  • 覆盖非最左边的基类的虚函数时，MSC++一般不创建调整块，也不增加额外的虚函数项。

虚继承

MSVC为了避免每次每次访问虚函数表都要进行虚基类表的查询操作，在派生类存在新的虚函数时，没有使用基类的虚函数表，而是在派生类中创建一个vfptr。
